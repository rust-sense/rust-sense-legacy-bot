# Inline LiamCottle Libraries Plan

**Date:** 2026-04-29
**Status:** Draft â€” Not yet executed

## Objective

Incorporate the code from `@liamcottle/push-receiver` and `@liamcottle/rustplus.js` directly into this codebase, fully converted to TypeScript. Use `protobufjs-cli` to compile `.proto` files into static JS modules at build time instead of loading them at runtime. Remove both external dependencies and their patches.

## Why

- Eliminates external dependencies that are unmaintained forks
- Allows fixing bugs (undefined `keys` variable, missing types, unhandled async errors)
- Gives full control over the FCM listener and Rust+ WebSocket protocol code
- Removes the need for `pnpm patch` on `rustplus.proto`
- Static proto compilation removes the need to copy `.proto` files to `dist/` and avoids the `__dirname`/ESM problem

### Bug Fixes During Port

1. **Undefined `keys` variable** in `Client._onDataMessage()` (line ~184 of `client.js`)
   - `message = decrypt(object, keys);` references an undefined `keys` variable
   - The bot does not use encrypted FCM notifications, so the simplest fix is to early-return if `crypto-key` is present and skip decryption entirely
   - Alternatively, remove the decrypt path and only handle unencrypted messages (which the current bot already relies on)

2. **Missing `.catch()` on async calls**
   - `checkIn()` inside `connect()` can throw (e.g., "interior hyphen" from corrupted credentials)
   - Add `.catch()` or `try/catch` around all async calls that could reject

3. **Module system conversion**
   - Convert all `require()` to ESM `import`
   - Convert `module.exports = class` to `export default class`
   - Add proper TypeScript types for constructor params, events, and public methods

4. **TypeScript types**
   - `Client` constructor: `(androidId: string, securityToken: string, persistentIds: string[])`
   - Events: `connect`, `disconnect`, `error`, `ON_DATA_RECEIVED`
   - `ON_DATA_RECEIVED` payload: `{ persistentId: string, appData: Array<{key: string, value: string}>, ... }`

5. **Replace dynamic proto loading with static imports**
   - `parser.js` and `gcm/index.js` call `protobuf.load(path.resolve(__dirname, '*.proto'))` then `proto.lookupType(...)` â€” drop this entirely
   - Instead import directly from the generated module:
     ```typescript
     import { mcs, checkin_proto } from './proto/mcs_pb.js';
     // Use mcs.LoginRequest, mcs.DataMessageStanza, checkin_proto.AndroidCheckinRequest, etc.
     ```
   - Remove all `protobuf.load()` calls and the `path`/`url` imports they required; no `__dirname` needed

### Bug Fixes During Port

1. **Replace dynamic proto loading with static imports**
   - `rustplus.js` calls `protobuf.load(path.resolve(__dirname, "rustplus.proto")).then(root => { this.AppRequest = root.lookupType("rustplus.AppRequest"); ... })` â€” drop this entirely
   - Instead import at the top of `RustPlus.ts`:
     ```typescript
     import { rustplus } from './rustplus_pb.js';
     ```
   - Replace each `root.lookupType("rustplus.X")` with `rustplus.X` directly; the `connect()` method no longer needs a `protobuf.load()` promise wrapper
   - This also eliminates the `__dirname`/ESM problem (no `path.resolve` needed at all)

2. **jimp v1.x API migration (required â€” not optional)**
   - The repo uses jimp v1.6.1; `camera.js` was written against v0.22. All three jimp calls need updating:

   | v0.22 API (camera.js) | jimp v1.x replacement |
   |---|---|
   | `new Jimp(width, height)` | `new Jimp({ width, height })` |
   | `Jimp.rgbaToInt(r, g, b, a)` | `jimp.rgbaToInt(r, g, b, a)` (instance or `@jimp/utils`) |
   | `image.setPixelColor(color, x, y)` | `image.setPixelColor(x, y, color)` (args swapped) |
   | `image.getBufferAsync(Jimp.MIME_PNG)` | `image.getBuffer('image/png')` |

   These are confirmed breaking changes and must be applied when porting `Camera.ts`.

3. **Request/response sequencing types**
   - `seq` counter starts at 1, increments per request
   - `sendRequest()` stores callback in a map keyed by `seq`
   - `sendRequestAsync()` wraps this in a Promise with timeout
   - Add proper types: `seq: number`, pending requests map, timeout handling

4. **EventEmitter generic types**
   - `connected`, `disconnected`, `error`, `message`, `request` events
   - Type `message` event payload as the deserialized protobuf response object

5. **Code style modernization**
   - Replace `var` with `const`/`let`
   - Add explicit return types on public methods
   - Convert `function` declarations to class methods

## Appendix â€” Proto Files Detail

### `mcs.proto` (FCM)

Defines the protocol spoken with `mtalk.google.com:5228`:
- `HeartbeatPing` / `HeartbeatAck` (TAG 0/1)
- `LoginRequest` / `LoginResponse` (TAG 2/3)
- `IqStanza`, `Close`, `StreamErrorStanza`, `StreamAck`, `SelectiveAck`
- `DataMessageStanza` (TAG 8) â€” contains `app_data`, `persistent_id`, `raw_data`, `from`, `category`

### `checkin.proto` / `android_checkin.proto` (GCM)

Defines the HTTP request to `android.clients.google.com/checkin`:
- `AndroidCheckinRequest` / `AndroidCheckinResponse`
- `ChromeBuildProto`, `AndroidCheckinProto`
- Device type enums (Android, iOS, Chrome Browser, Chrome OS)

### `rustplus.proto` (Rust+)

Defines the companion app protocol. Key messages:
- `Vector2`, `Vector3`, `Vector4`, `Half3`
- `ClanInfo`, `ClanLog`, `ClanInvitations`, `ClanActionResult`
- `AppEntityType` enum (Switch, Alarm, StorageMonitor)
- `AppMarkerType` enum (Player, Explosion, VendingMachine, CH47, CargoShip, etc.)
- `AppRequest`, `AppMessage`, `AppResponse`, `AppBroadcast`
- `AppInfo`, `AppTime`, `AppMap`, `AppTeamInfo`, `AppTeamMessage`
- `AppCameraSubscribe`, `AppCameraInput`, `AppCameraInfo`, `AppCameraRays`

**Important:** This must be the *patched* version that adds optional fields and new enums. Copy from the current patched `node_modules/@liamcottle/rustplus.js/rustplus.proto`, not from the upstream git repo.

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Generated proto modules not emitted to `dist/` | Low | High | Set `allowJs: true` in tsconfig so tsc includes and emits the generated `.js` files alongside compiled TypeScript. |
| Static API differences from dynamic `lookupType` | Low | Medium | The static module exposes the same `create`, `encode`, `decode`, `toObject` methods â€” the call sites change only at the top-level type reference (`mcs.LoginRequest` vs `proto.lookupType('mcs.LoginRequest')`). |
| TypeScript types don't cover all edge cases | Medium | Medium | Start with broad `any` types for internal plumbing, gradually narrow. The `.d.ts` files already model the public API. |
| Camera.ts jimp API changed between v0.22 and v1.6 | **High** | Medium | **Confirmed breaking changes** â€” see jimp v1.x migration table in Part 2 Bug Fixes. All three jimp call sites require updates. |
| File size increase | Low | Low | ~1,600 lines of TS added, ~400 lines of generated JS added, removes 2 external deps. Net positive. |
